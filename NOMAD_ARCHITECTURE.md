# NOMAD Architecture Strategy for AVL FIRE M Data

This document outlines the architectural decisions and actionable steps for migrating the AVL FIRE M CFD data pipeline into a fully-fledged NOMAD Research Data Management (RDM) plugin.

## Architecture Overview

We are building a highly advanced, automated data harvesting pipeline for CFD data, moving away from traditional manual file uploads.

### 1. Data Ingestion (The Harvesting Parser)
* **Trigger:** Users upload a simple configuration file (e.g., `harvest.yaml`) containing the `project_directory`, `model_name`, and `case_name`.
* **Action:** A NOMAD parser runs on the server. It uses `paramiko` to SSH into the simulation cluster, locate the results via `utils.retrieve_avl_fire_data_paths`, and download the raw data securely using environment variables for credentials.

### 2. 3D Data Handling (EnSight -> XDMF/HDF5)
* **Conversion:** The parser runs `ensight_to_xdmf.py` to convert the messy EnSight format into standard XDMF/HDF5.
* **Storage:** EnSight files are discarded. The XDMF/HDF5 files are saved to NOMAD's **Raw File Storage**. This keeps them in a standard format so researchers can download them and immediately open them in ParaView.
* **Visualization:** The parser uses `visualize_3d_data.py` (via `pyvista`) to extract 2D slices. These slices are serialized into Plotly JSON and stored in the NOMAD archive metadata for instant, interactive 3D/2D browser previews.

### 3. 2D Data Handling (.csv timeseries)
* **Integrity:** The original 2D `.csv` files generated by the solver (averages, mass flows) are preserved to retain temporal resolution and solver fidelity.
* **Parsing:** The headers are normalized using `firem_name_parser_integration.py`.
* **Storage:** Data arrays are assigned to NOMAD `Quantity` definitions. NOMAD automatically handles writing these arrays to its highly efficient internal HDF5 archive.

### 4. Metadata Handling (.asix inputs)
* **High-level search:** Critical parameters (e.g., inlet velocities, turbulence models) are parsed into explicitly defined NOMAD quantities (`Quantity(type=np.float64)`) to populate the Elasticsearch database.
* **Complete record:** The entire parsed `.asix` JSON is dumped into a catch-all `Quantity(type=dict)` so the NOMAD web UI renders a collapsible tree menu of every simulation setting.

## Actionable Next Steps

### Step 1: Bootstrap the Plugin Repository
1. Clone the `nomad-distro-template` (https://github.com/FAIRmat-NFDI/nomad-distro-template).
2. Set up `pyproject.toml` to declare the package (e.g., `nomad-avl-fire`) and register the plugin entry points for the Schema and Parser.
3. Move the existing `src/` directory, `visualize_3d_data.py`, and `transfer_and_convert_3d_data.py` into the plugin's source tree.

### Step 2: Define the Data Model (`schema.py`)
Write the NOMAD `EntryData` classes. You'll need sections for:
* Simulation Metadata (from the `.asix` file).
* Averaged Results (to hold the 2D CSV arrays).
* Visualization (a dictionary quantity to hold the Plotly JSON).
* File References (a string quantity linking to the downloaded XDMF mainfile).

### Step 3: Create the Harvesting Parser (`parser.py`)
1. Create a class inheriting from `MatchingParser` and configure it to match a specific file extension (e.g., `.fire_harvest.yaml`).
2. Inside the `parse()` method, read the YAML file to get the target parameters.
3. Import and call the `paramiko` fetch routine, ensuring SSH credentials are read from the NOMAD server's environment variables (`os.getenv`).
4. Call the `ensight_to_xdmf` conversion function.

### Step 4: Connect the Pieces in the Parser
Once the files are downloaded and converted locally in the NOMAD worker's directory:
1. Parse the `.asix` file and populate the schema metadata.
2. Read the `.csv` file, process it with `firem_name_parser_integration.py`, and map it to schema quantities.
3. Run the Plotly extraction logic from `visualize_3d_data.py` and assign the JSON output to the schema's visualization quantity.

### Step 5: Local Testing
Use the `nomad parse` CLI command locally to test the parser against a dummy `.yaml` file, ensuring it connects, downloads, processes, and outputs the expected NOMAD archive JSON.
